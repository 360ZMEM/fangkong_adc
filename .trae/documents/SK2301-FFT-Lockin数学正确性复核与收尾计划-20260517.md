# SK2301 FFT/Lock-in 数学正确性复核与收尾计划

## Summary

本计划基于当前仓库的实际代码状态，目标不是从零重构，而是完成一次“数学正确性复核 + 剩余缺口收尾”。

本次计划确认并固化以下三条约束：

1. `FFT`、`Lock-in`、波形 X 轴的样本间隔只允许来自硬件采样率 `Fs`，不得由 `time.time()`、`time.monotonic()` 或任何本地时间反推。
2. `PipelineWorker` 必须以固定长度环形缓冲区为基础，并按固定 `hop_size_samples` 触发 DSP，而不是“收到多少算多少”。
3. 协议级丢包检测以自动上传模式的 `pack_num` 为正统实现；主动读模式若协议头没有 `pack_num`，必须明确暴露该能力边界。

本计划同时覆盖两种传输模式：

- `poll`：主机主动读寄存器 `19`
- `auto_upload`：设备自动上传 `52` 字节波形包头

用户既有决策沿用如下：

- 两种模式都要规划与支持
- 缺包补点默认采用 `zero_order_hold`

## Assumptions & Decisions

### 核心决策

1. `sample_rate_hz` 的合法数学来源限定为：
   - `poll` 模式：`config.device.sample_rate_hz`
   - `auto_upload` 模式：上传头 `sample_rate_hz`，若缺失则回退到当前已知 `Fs`
2. `time.time()` 与 `time.monotonic()` 允许继续用于：
   - 日志/事件时间戳
   - 文件命名
   - 网络调度
   - 接收速率和延迟诊断
3. 上述本地时间绝不允许参与：
   - 波形 X 轴生成
   - `FFT` 频率轴
   - `Lock-in` 参考相位
   - 样本间隔 `Δt` 推导
4. `poll` 模式能力边界固定为：
   - 保证固定 `Fs`、固定窗口、固定 hop 的数学正确性
   - 不承诺协议级 `pack_num` 丢包检测
5. `auto_upload` 模式能力边界固定为：
   - 支持 `pack_num` 连续性校验
   - 支持 `zero_order_hold` / `zero_padding`
   - 默认补点策略为 `zero_order_hold`
6. `scope_total_window_ms=200`、`scope_div_ms=20` 继续作为默认示波器显示基准。

### 非目标

- 不重新设计 UI 布局
- 不更改设备端协议
- 不假定主动读模式存在尚未证实的隐藏 `pack_num`
- 不把日志/存盘的 wall-clock 时间戳删掉
