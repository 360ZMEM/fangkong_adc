# [OPEN] debug-sk2301-ui-jitter

## 背景

- 设备：`SK2301`
- 当前连接地址：`192.168.1.198`
- 现象 1：绘图窗口因横向长度变化出现抽搐
- 现象 2：三个通道颜色一致，难以区分
- 目标：先修复这两个 UI/绘图问题，再启动上位机辅助观察是否还有其他真机问题

## 假设

1. `WaveformPlot.update()` 每次传入的数据长度变化，导致 `PlotWidget` 自动重新计算 X/Y 范围，从而引发窗口抽搐。
2. GUI 刷新过程中没有固定绘图窗口宽度，`snapshot.waveform` 点数在积累期持续增加，造成 `setData()` 输入长度抖动。
3. `pyqtgraph` 曲线没有显式指定 `pen`，默认颜色循环在当前实现中没有生效或被重复复用，导致三通道显示为相同颜色。
4. 频谱图与波形图都开启了自动范围调整，二者在 30Hz 刷新下不断重排布局，叠加形成“抽搐”观感。
5. 真机返回流量或采样状态存在异常，导致 UI 实际问题不止是绘图，还可能伴随队列波动、解析错位或状态切换不稳，需要运行时证据确认。

## 计划

1. 阅读 GUI 绘图相关代码，定位波形和颜色配置点。
2. 先加入最小运行时插桩，只记录绘图数据长度、自动范围行为和通道颜色选择。
3. 运行上位机复现真机现象并分析证据。
4. 基于证据做最小修复。
5. 重新运行并对比修复前后的表现。

## 当前状态

- 状态：进行中
- 已完成 UI 抽搐与颜色区分修复。
- 已完成真机慢滚动问题定位与修复。

## 真机证据

- 真机 `read_stream(1408)` 返回的 `channel_id` 只有 `[0, 1, 2]`，说明设备当前只回激活通道，不是 16 通道全回。
- 原逻辑按 16 通道解码时，每包 `1408` 字节只得到 `22` 个时间点，并出现大规模 `channel_mismatch_count`。
- 真机探测中，同一包按 3 通道解释可得到 `117` 个时间点，且通道序列与 `0,1,2` 循环一致。
- `1408` 不是 3 通道单帧 `12` 字节的整数倍，会导致跨包相位轮转；改成 `1404` 后每包正好 `117 * 12` 字节，跨包稳定。
- 真机单包周期约 `58ms`，折算约 `15.4 包/s`，对应约 `1800 样本/s`，远低于 USB2.0 以太网链路上限，因此不是链路带宽瓶颈。

## 结论

1. 主因是协议解释与当前配置不匹配：设备处于 3 通道模式，但代码仍按 16 通道整帧解码。
2. 次因是请求字节数未按 3 通道整帧对齐，导致跨包边界相位漂移。
3. 额外风险是读流泵每 `1ms` 发送一次请求，远快于设备出数节奏，会制造不必要的请求积压。

## 已实施修复

- 波形图固定 3 秒时间窗，禁用 X 轴自动缩放。
- 三通道显式指定颜色。
- 左侧状态栏宽度固定，避免 `Lock-in` 文本长度变化引发布局抖动。
- 新增按激活通道序列重组的 `ActiveChannelDecoder`，支持跨包保留残样本。
- `read_bytes_per_request` 默认从 `1408` 调整为 `1404`。
- 读流请求自动按激活通道字节数对齐，并按样本产生速率节流，不再 `1ms` 无脑发送。
