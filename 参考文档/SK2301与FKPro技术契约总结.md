# SK2301 与 FKPro 技术契约总结

本文档面向“透明以太网指令的 16 通道 24 位 ADC 上位机程序”前置设计，综合以下两份手册及其 MinerU 解析稿整理：

- `参考文档/SK2301一体化信号采集控制模块产品说明书_V2.0.pdf`
- `参考文档/MinerU_markdown_SK2301一体化信号采集控制模块产品说明书_V2.0_2055862219376885760.md`
- `参考文档/FKPro通讯协议及编程说明书_V1.2.pdf`
- `参考文档/MinerU_markdown_FKPro通讯协议及编程说明书_V1.2_2055862315048960000.md`

目标不是逐页复述，而是提炼后续开发必须遵守的“设备能力契约 + 协议契约 + 实现约束”。

## 1. 结论先行

对后续软件开发最重要的结论如下：

1. 官方建议以太网编程优先使用 `FkPro`，而不是 Modbus。
2. 以太网访问本质上是基于 `socket` 的二进制收发。
3. 默认设备网络参数为：
   - IP：`192.168.1.198`
   - 端口：`1600`
   - 网关：`192.168.1.1`
   - 子网掩码：`255.255.255.0`
4. FkPro 核心只有四类命令：
   - `0x7A01` 常规寄存器读
   - `0x7A10` 常规寄存器写
   - `0x7A02` 串行寄存器读
   - `0x7A20` 串行寄存器写
5. 所有字段都按大端序排列；以太网场景下 `inst_addr = 0x0000`，`crc = 0x0000`。
6. 24 位采样数据不是 3 字节，而是 `4` 字节一个样本：`高8位 + 中8位 + 低8位 + 通道号`。
7. 串行寄存器单次返回最多 `1440` 字节，因此 16 通道 24 位场景必须按包循环读取，并考虑整帧对齐。

## 2. SK2301 设备能力契约

### 2.1 产品定位

SK2301 是一体化信号采集控制模块，面向设备健康监测和预测性维护场景，可直接接入振动、温度、转速、压力、位移、开关等传感器，并通过以太网实时上传或由上位机主动读取。

### 2.2 与本项目最相关的高速采集能力

手册给出的高速采集关键参数如下：

- 高速采集通道数：`8/16` 路可选
- 高速采样方式：同步采集
- 高速采样分辨率：`24bit`
- 单通道采样速率：`1Hz ~ 128KHz/ch`
- 支持的采样率不是任意值，而是固定 `24` 个档位：
  - `1, 5, 10, 20, 40, 50`
  - `100, 200, 250, 400, 500, 800`
  - `1000, 2000, 4000, 6400, 8000, 12800`
  - `16000, 25600, 32000, 51200, 64000, 128000`
- 总采样速率 `<= 256KSPS` 时采用实时传输模式
- 总采样速率 `> 256KSPS` 时采用高速缓存模式

### 2.3 输入类型和量程

高速通道支持以下类型：

- IEPE：24V 供电，4mA 工作电流，默认调理带宽 `0.5Hz ~ 1MHz`
- 电压：支持 `±10V`、`±5V`、`±2.5V`，默认 `±10V`
- 电流：支持 `0~20mA / 4~20mA`，默认采样电阻 `240Ω`

这意味着后续程序至少要把“通道配置”抽象成：

- 通道是否使能
- 通道类型（IEPE / 电压 / 电流）
- 量程
- 滤波配置
- 采样率

### 2.4 附加功能能力边界

SK2301 不是单一固定配置，而是按型号组合：

- 低速电压/电流采集：最多 `8` 路，24bit，`1~10Hz/ch`
- PT100：最多 `6` 路，`-50~200°C`
- D/A 输出：`8` 路，16bit，默认 `0~10V`
- 光耦隔离输入：最多 `8` 路
- 继电器输出：最多 `8` 路
- 计数器/频率计/A/D 触发：最多 `2` 路
- TF 卡：`16~256GB`
- RS485：`1` 路

因此后续软件架构不能把设备硬编码为“只有 16 路高速 ADC”，而应允许：

- 读取并识别当前设备实际功能组合
- 按功能存在与否启用对应寄存器和界面

### 2.5 型号与订货约束

型号格式为：

```text
SK2301-A□□-B□□-C□□-D□□-E□□-F□□-G□□
```

关键约束：

- `A + B + C <= 16`
- 出厂时 A/B/C 通道按照 IEPE -> 电压 -> 电流 的顺序排列
- 用户不能任意调整该排序
- 只有 A/B/C 这部分高速采集通道可做时域/频域分析

这意味着上位机最好把“逻辑通道号”和“通道物理类型映射”分开管理。

## 3. 网络与联机契约

### 3.1 默认网络参数

产品说明书明确给出默认网络配置：

- IP：`192.168.1.198`
- 端口：`1600`
- 网关：`192.168.1.1`
- 子网掩码：`255.255.255.0`

联机前要求上位机电脑先切到同一网段。

### 3.2 通讯方式

SK2301 支持：

- 以太网 `10/100M` 自适应
- 协议支持 `TCP/UDP、ICMP、DHCP`

但在软件开发章节中，官方明确推荐：

- 采用 `FkPro` 协议
- 基于 `socket` 编程实现连接、设置、读数
- 编程语言不限，只要能建立 socket 连接

### 3.3 官方建议的开发顺序

手册建议按以下流程进行：

1. 先用《以太网调试助手》联通设备。
2. 用调试工具手工发送指令，熟悉返回包。
3. 确认每条命令的读写地址和数据区含义。
4. 再把这些报文嵌入上位机程序。

这个建议应直接固化为开发流程，而不是跳过联调直接写代码。

## 4. FkPro 协议契约

### 4.1 协议族定位

同系列产品通常支持三种协议：

- `FkPro`
- `Modbus_Tcp`
- `Modbus_Rtu`

官方建议在高速数据交互场景下优先选 `FkPro`，因为它更适合高吞吐数据传输。

### 4.2 读写对象模型

FkPro 把可操作对象分成两类：

- 常规寄存器：16bit 宽，适合配置和状态量
- 串行寄存器：字节流接口，适合大块数据传输

因此后续程序里必须有两套访问模型：

- `read/write_registers()`
- `read/write_stream_register()`

### 4.3 通用帧格式

#### 读命令

```text
protocol_code(4) + inst_addr(2) + cmd_code(2) + reg_addr(2) + data_num(2) + err_type(2) + crc(2)
```

#### 写命令

```text
protocol_code(4) + inst_addr(2) + cmd_code(2) + reg_addr(2) + data_num(2) + err_type(2) + crc(2) + data...
```

字段语义：

- `protocol_code`：固定 `0x464B666B`，即 `FKfk`
- `inst_addr`：
  - 以太网/WIFI/GPRS 场景不用，固定为 `0`
  - 485/232/ZigBee 场景才有设备地址意义
- `cmd_code`：四种命令码之一
- `reg_addr`：起始寄存器地址
- `data_num`：
  - 常规寄存器读写时表示寄存器个数
  - 串行寄存器读写时表示字节数
- `err_type`：错误码/扩展状态
- `crc`：
  - TCP/UDP 场景通常不校验，填 `0`
  - 其他链路需要按手册要求计算

### 4.4 协议层必须遵守的约束

1. 所有字段大端序。
2. 所有常规寄存器宽度为 `16bit`。
3. 单次串行数据区最大 `1440` 字节。
4. 不允许越地址操作。
5. 指令必须串行发送，一问一答，不要并发发包。

## 6. 24 位采样数据解析契约

### 6.1 采样数据排列方式

手册对寄存器 `19` 的说明非常关键：

- 采样数据在 `data[data_num]` 中按采样顺序依次排列
- 同步采集时，每个通道相同序号的样本对应同一个时刻
- 16bit 设备每样本占 `2` 字节
- 24bit 设备每样本占 `4` 字节

24 位场景的数据布局为：

```text
[高8bit][中8bit][低8bit][通道号]
```

### 6.2 通道号规则

- 通道号从 `0` 开始
- 第 1 通道的通道号为 `0`
- 第 2 通道的通道号为 `1`

因此解析时不能只按“每 4 字节转一个整数”，还必须校验尾字节通道号是否符合预期轮转。

### 6.3 电压/电流换算公式

对于 24 位产品（SK2300 系列），手册给出如下公式：

- `±10V`：`V = 10.0 * Vhex / 0x80000000`
- `±5V`：`V = 5.0 * Vhex / 0x80000000`
- `±2.5V`：`V = 2.5 * Vhex / 0x80000000`
- 电流：`mA = V * 1000 / R`，默认 `R = 240Ω`

实现时要注意：

- `Vhex` 应视为有符号 32 位整数
- 上位机最好同时保留“原始整数值”和“物理量值”

### 6.4 包大小建议

虽然协议允许一次最多读 `1440` 字节，但对 `16` 通道 `24bit` 来说：

- 每个时间点占 `16 × 4 = 64` 字节
- `1440 / 64 = 22.5`

也就是说 `1440` 不能整除一整帧 16 通道数据。

建议：

- 单次读取 `1408` 字节，即 `22` 个完整时间点
- 或者在解析器里支持“残帧缓存拼包”

如果首版目标是先稳定拿到完整波形，优先选“整帧对齐读取”。

## 7. 自动上传契约

FkPro 不仅支持主机主动读，还支持设备主动上传。

### 7.1 自动上传总体特点

- 默认推荐用 `UDP`
- 自动上传包头中自带包序号和校验字段
- 支持心跳包与波形数据包
- 支持秒同步、事件号和多包拼接

### 7.2 心跳包关键字段

- `pack_type = 0x01`
- 时间字段采用年月日时分秒
- `event_num` 固定为 `0`

### 7.3 波形自动上传关键字段

- `pack_type = 0x02`
- `pack_code = 0x01` 表示高速采样数据包
- `pack_num`：包序号
- `event_num`：事件序号，用于同一事件多包拼接
- `time`：相对 `1970-01-01 00:00:00` 的秒数
- `ad_fre`：采样频率
- `channel_en`：通道使能
- `sec_sync`：秒同步标记

对当前“透明以太网指令”项目来说，第一阶段建议先不做自动上传解析，优先完成“主机主动读寄存器/读波形”。

## 8. 建议的软件分层

### 8.1 传输层

职责：

- TCP 连接建立/断开
- 超时控制
- 收发互斥
- 自动重连

关键要求：同一时刻只允许一个请求在飞。

### 8.2 FkPro 编解码层

职责：

- 构造读写帧
- 大端序编码/解码
- 常规寄存器和串行寄存器的统一封装

建议 API：

- `read_regs(addr, count)`
- `write_regs(addr, words)`
- `read_stream(addr, size)`
- `write_stream(addr, data)`

### 8.3 设备服务层

职责：

- 初始化检查
- 采样参数配置
- 启停控制
- 状态轮询
- 波形读取
- TF 文件读写
- 网络参数维护

### 8.4 数据解析层

职责：

- 24 位样本解包
- 通道号校验
- 原始值转物理量
- 时间戳和秒同步处理
- 按通道重组矩阵

### 8.5 业务层

职责：

- 连续采集任务管理
- 数据落盘
- 可视化/监测/告警
- 面向后续透明指令封装

## 9. 推荐的首版实现路径

针对“16 通道 24 位 ADC 透明以太网指令程序”，推荐按下面顺序落地：

1. 实现 TCP 连接到 `192.168.1.198:1600`
2. 实现 FkPro 四种命令的编码与解码
3. 先读 `Init_Status(0)`，确认设备初始化完成
4. 实现“写寄存器 `10-16` 设置采样参数”
5. 实现“写寄存器 `17` 启动采集”
6. 实现“读寄存器 `18` 查询状态”
7. 实现“读串行寄存器 `19` 获取波形数据”
8. 完成 24 位、16 通道数据重组和物理量换算
9. 再补充 TF 模式、文件操作、网络参数配置、RTC 配置

如果首版目标是“稳定拿到 16 通道波形”，建议只做：

- 模式 0 实时采集
- 固定采样率
- 固定量程
- 固定 TCP 传输
- 主机主动读波形

先把主链路跑通，再逐步扩展。

## 10. 当前仍需确认的现实问题

虽然手册足够支持首版开发，但真正编码前仍建议确认以下问题：

1. 当前设备是否确实是 `16` 路高速通道全配
2. 16 路中每路的实际类型是 IEPE、电压还是电流，是否混布
3. 量程默认是否为 `±10V`，有没有定制成 `±5V` 或 `±2.5V`
4. 是否需要秒同步时间信息
5. 是否要支持 TF 卡离线采集
6. 是否后续需要设备主动 UDP 上传

这些不会改变协议骨架，但会影响默认寄存器配置、数据换算和界面设计。

## 11. 作为后续开发契约的落地约束

后续所有代码建议默认遵守以下规则：

1. 默认协议：`FkPro over TCP`
2. 默认目标：`192.168.1.198:1600`
3. 默认字节序：大端
4. 默认设备地址：`0x0000`
5. 默认 CRC：TCP 场景填 `0x0000`
6. 默认 24 位数据按 4 字节样本处理
7. 默认单次读波形按“整帧对齐”策略，不盲目拉满 `1440` 字节
8. 默认命令采用严格串行的一发一收模型

只要首版程序严格遵守以上 8 条，基本就落在官方文档允许且可联调的路径上。
